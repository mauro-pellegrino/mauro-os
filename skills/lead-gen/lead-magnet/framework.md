## What This Subtype Is

A framework for a content or acquisition system. The magnet teaches the agency owner how to think about and execute a specific system (content engines, lead magnets, call-to-content pipelines, inbound playbooks). Higher abstraction than a prompt pack, lower specificity than a case study.

Subtype identity: systems thinking, memorable mental model, repeatable production rules.

**Track record note:** the framework structure below was proven on the agency account Mauro ran before this brand (framework magnets were consistent top performers there). Shipped magnets for this account live in `brand/lead-magnets/`; log each one's performance as its own data accrues.

## Input Form

Before drafting, get these from Mauro:

1. **System being framed** (content engine, lead magnet system, call-to-content pipeline, inbound measurement loop, etc.)
2. **Number of components** in the framework. **5 is the rule.** 5 pages or 5 components is the proven baseline. 5 feels substantial enough to earn the autodm without losing the reader's attention. Flex to 4 or 6 only if the topic truly demands.
3. **Component naming convention** (pillars / phases / parts / structures / layers — pick one consistent label)
4. **Existing internal version of the framework** (Notion page, doc, or just Mauro's head)
5. **Worked examples or visual demonstrations** required for the asset (yes by default)
6. **Target reader tier** (informs the subtitle and authority anchor; default is the established agency owner per `brand/audience.md`)
7. **Trigger keyword** (3-7 chars, ALL CAPS, topic-related. Examples: ENGINE, INBOUND, SYSTEM)
8. **Asset format** (Notion 5-pager, Gamma deck, Canva slides, LinkedIn doc — what's the end material)
9. **Before/after illustration** (powerful for frameworks that flip an old way of working, e.g. outreach-dependent pipeline before vs measured inbound engine after)

If the framework doesn't have a memorable name yet, propose 2-3 names before drafting. Name it in 1-3 sticky words ("The AI Content Engine" style, named as Mauro's own system).

## Asset Structure

The magnet ships as a Notion page, Gamma deck, Canva slides, or LinkedIn doc.

### Title format

`[Catchy framework name or outcome] for [Audience / Context]`

Example patterns for this brand's lane:
- "The AI Content Engine for Agency Owners (2026)"
- "The Agency Inbound Playbook (NOT more posting)"
- "Turning Client Calls into Booked Calls in 2026"

Titles always anchor on either a specific outcome, a year, or a contrarian stance ("NOT more posting"). Generic titles ("How to write content") do not ship.

### Subtitle / opening framing

`The ideal [system] to get inbound clients in [year] with [specific differentiator]`

or

`Why a [Quality X] Process Should Be Your Priority`

The opening framing earns the reader's attention before diving into the framework.

### Body structure

```
[Framework name + 1-sentence pitch]

PART 1: [Component name]
  [Definition / what it is in 2-3 lines]
  [Why it works / mechanism]
  [Worked example or visual]

PART 2: [Component name]
  ...

[Repeat for 4-6 parts total]

CLOSING SECTION: [Where this fits in the bigger picture / what to do next]
```

Example component split for a content-engine framework (illustrative, adapt to the actual system):
- Source (the real work: client calls, builds, results)
- Extraction (AI turns the source into raw material)
- Formats (posts, articles, lead magnets, long-forms from one source)
- Distribution (channels, cadence, auto-DM)
- Measurement (content to booked calls, the weekly loop)

Each component gets: definition, production structure, when to use it, real example.

### Rules for each component

- Named in 1-3 words (memorable, not descriptive)
- Defined in 2-3 lines max (definitions live in the asset, not the post)
- Mechanism explained (why this component works, grounded in the real engine, not theory)
- Worked example or visual reference for every component
- Where it fits in the broader system (don't leave components floating)

### Before/After illustration (optional but powerful)

When the framework flips an old way of working, include a before/after section:

```
BEFORE
- [Old behavior 1, e.g. posting when inspiration strikes]
- [Old behavior 2, e.g. pipeline runs on referrals]
- [Old behavior 3, e.g. no idea what converts]

AFTER
- [New behavior 1, e.g. every client call becomes content]
- [New behavior 2, e.g. inbound calls booked weekly]
- [New behavior 3, e.g. a Monday number that tells you what worked]
```

Use this for any framework positioned as "the new playbook" or a break from the default way agencies handle their own marketing.

## Post Hook Formulas (Subtype-Specific)

`_master.md` has the universal hook formulas. For framework subtypes specifically:

### X hooks
- **"Your [thing] [does problem], and that's [killing X]"** (e.g. "Your agency's content looks like everyone else's, and that's killing your inbound.")
- Less commonly the tool-led "[Tool] is INSANE" formula (frameworks aren't always tool-anchored)

### LinkedIn hooks
- **"[Component] is the 80/20 of [thing]"** (e.g. "Lead magnets are the 80/20 of agency inbound.")
- **"You need to [contrarian action]"** (e.g. "You need to build content that doesn't look like content.")
- **"[Thing] requires a NEW playbook"** (e.g. "Agency inbound in 2026 requires a NEW playbook.")

### Setup line

The framework post body usually includes a stakes line in the setup paragraph, anchored to Mauro's own proof (see `_master.md` authority anchors, sign-off required):

- "I run this exact engine daily for a real B2B agency. Here's the framework behind it..."
- "[ANCHOR — signed-off proof line]. I broke the system into 5 components..."

The setup positions the framework as earned, not theoretical.

### Bullet structure (the "Inside the guide:" list)

- The full breakdown of each [component] and why it works
- The exact [structure / process] I use for each one
- How each [component] maps to different [scenarios] so you know when to use what
- Real examples from the engine I run

The bullets always include: a breakdown + a process + a mapping rule + real examples. That's the 4-part promise.

## Anti-Patterns (Specific to This Subtype)

- Generic framework names ("The Content Scaling System") — needs to be specific or contrarian to be memorable
- Components without mechanisms — "what" without "why" reads as a listicle, not a framework
- Component count outside 4-6 (5 is the proven baseline) — past 6 readers stop tracking, fewer than 4 feels thin
- Skipping worked examples — frameworks without examples sound theoretical
- Hook formulas without setup — the hook works because it's followed by a proof anchor in paragraph 2. Skipping the anchor weakens the hook.
- Treating this like a prompt pack — frameworks are systems thinking, not paste-ready commands
